# 02_小红书热点追踪工作流

## 目标

基于 Feishu 关键词池，定时触发小红书搜索 UI Flow，获取前 N 条可见结果，形成 evidence JSON 和 normalized records，并写入 PostgreSQL / 回写飞书热点池 / 供 Coze 分析。

## V2 基本链路

```text
Feishu 关键词池
-> n8n 定时任务
-> n8n 生成 search_job
-> browser-worker 接收任务
-> Provider Router 选择 kuaijingvs_yingdao_rpa
-> 跨境卫士打开 account_id 对应账号/店铺环境
-> 影刀 RPA 执行小红书搜索 UI Flow
-> 影刀截图并输出 search_evidence.json
-> browser-worker 读取 evidence JSON
-> PostgreSQL 写入 evidence 和 normalized_records
-> n8n 回写 Feishu 热点池
-> Coze/Dify 可选做热点分析
```

## browser-worker 职责

- 接收 `SearchJob`。
- 选择 provider。
- 调度跨境卫士和影刀。
- 读取 evidence JSON。
- 返回 WorkerResult。
- 写入 PostgreSQL。

browser-worker 不再作为主链路直接打开小红书搜索页、输入关键词和提取 DOM。`selenium_chrome` 和 direct search URL 只保留为本地 debug。

## 影刀 RPA 职责

- 在真实浏览器环境中执行搜索 UI Flow。
- 输入关键词，等待页面结果。
- 截图留证。
- 提取前 N 条页面可见结果。
- 输出 `search_evidence.json`。
- 遇到登录、验证码、扫码、安全确认、风控或账号限制时停止并输出 `waiting_human_verification`。

## 需要记录的字段

- job_id
- keyword
- account_id
- provider_type
- captured_at
- rank
- title
- author
- published_at_text
- note_id
- note_url
- metric_raw_text
- like_count_text
- screenshot_path
- evidence_json_path
- status
- error_code
- error_message

## 排名判断原则

- 以影刀在真实浏览器页面看到的展示顺序为准。
- 每条结果记录 rank。
- 如果页面出现广告、推荐、置顶等特殊标记，后续应记录 result_type。
- 页面异常必须截图并写入 evidence。

## 截图策略

第一阶段：

- 搜索结果页截图。
- 前 N 条结果结构化提取。
- 异常时保存错误截图和 error evidence。

后续阶段：

- 对重点结果进入详情页截图。
- 增加正文、标签、评论、收藏、分享等更细字段。

## 禁止事项

- 不调用小红书未授权接口。
- 不逆向 request。
- 不伪造 XHR、fetch、token、cookie、sign。
- 不绕过登录、验证码、扫码、安全确认、风控或账号限制。
