# 08_整体开发框架与开发路线图

## 1. 文档定位

本文是 `xhs-automation` 项目的总体开发框架与路线图。它低于 `AGENTS.md` 的强制规则，高于临时讨论。小红书自动化相关任务应同时参考 `docs/09_小红书自动化工作流_RPA重构版_V2.md`。

## 2. 当前主架构

V2 主链路已经从“browser-worker 直接操作小红书页面”调整为：

```text
Feishu
-> n8n
-> browser-worker
-> Provider Router
-> KuaJingVS / 跨境卫士启动账号环境
-> Yingdao / 影刀 RPA 执行小红书 UI Flow
-> evidence JSON
-> PostgreSQL / Feishu / Coze
-> OpenClaw 人工协作
```

## 3. 开发边界

- 小红书搜索和发布主链路必须通过真实浏览器页面中的 UI Flow 完成。
- 主链路由跨境卫士提供账号/店铺环境，由影刀 RPA 执行页面动作。
- browser-worker 不再作为直接页面操作者，而是 RPA 调度器和 evidence 数据承接器。
- `selenium_chrome_provider` 仅保留为本地 debug，用于验证 FastAPI、截图、selector、evidence 格式。
- direct search URL 仅允许 debug，不作为主链路。
- 禁止调用小红书未授权接口。
- 禁止逆向 request。
- 禁止伪造 XHR、fetch、token、cookie、sign。
- 禁止绕过登录、验证码、扫码、安全确认、风控或账号限制。

## 4. 组件职责

### Feishu

运营工作台，维护关键词池、热点池、发布池、人工审核状态和异常备注。

### n8n

任务编排层，负责从 Feishu 读取数据、生成 job、调用 browser-worker、接收结果和回写 Feishu。

### browser-worker

RPA 调度器 + evidence 数据承接器。负责接收 job、选择 provider、调用跨境卫士和影刀、读取 evidence JSON、返回 WorkerResult，并将 normalized records 写入 PostgreSQL。

### Provider Router

根据 `provider_type` 路由：

- `selenium_chrome`：本地 debug-only。
- `yingdao_rpa`：影刀 RPA 执行。
- `kuaijingvs_yingdao_rpa`：跨境卫士环境 + 影刀 RPA，后续主链路。

### KuaJingVS / 跨境卫士

账号/店铺环境启动器，负责 account_id 到 profile/environment 的映射和环境生命周期。

### Yingdao / 影刀 RPA

小红书 UI Flow 执行器。执行搜索、发布、截图、结果提取，并输出 evidence JSON。

### PostgreSQL

结构化底账，保存 evidence 头信息、normalized search records、发布结果和执行日志。

### MinIO

证据和素材仓库，保存截图、页面证据、图片素材、error.json。

### Coze / Dify

智能分析和内容生成层，消费 normalized records，不直接控制浏览器。

### OpenClaw

人工协作入口，通知 waiting_human_verification、failed、success，并支持后续重试/继续。

## 5. 标准数据接口

`evidence JSON` 和 `normalized_records` 是后续统一数据接口：

- 影刀负责输出 evidence JSON。
- browser-worker 读取 evidence JSON。
- PostgreSQL 写入 evidence 和 normalized records。
- Feishu / Coze / Dify 消费 normalized records。

## 6. 路线图

### 阶段 1：browser-worker 基础骨架

已完成 FastAPI、schema、job registry、日志、错误码、本地 debug provider。

### 阶段 2：debug-only 页面验证

已用 `selenium_chrome_provider` 验证本地搜索、截图、items 清洗、evidence JSON、normalized records 和 PostgreSQL 可选写入。

### 阶段 3：RPA 重构

目标是把主链路切到：

```text
browser-worker -> Provider Router -> KuaJingVS -> Yingdao RPA -> evidence JSON
```

建议任务：

1. Task 24A：Provider Router skeleton。
2. Task 24B：YingdaoService skeleton。
3. Task 24C：KuaJingVSOpenAPI skeleton。
4. Task 24D：`/api/xhs/search` 切到 provider router，Selenium 保留 debug-only。

### 阶段 4：底账和回写

- PostgreSQL 保存 evidence 和 normalized records。
- MinIO 保存截图和证据文件。
- n8n 回写 Feishu 热点池/发布池。
- Coze/Dify 消费热点记录。

## 7. Codex 使用规则

- 每次只完成一个明确 Task。
- 修改数据库结构必须提供 schema 或 migration。
- 文档、代码、测试要小步推进。
- 涉及小红书页面执行时必须遵守 RPA 主链路和禁止事项。
