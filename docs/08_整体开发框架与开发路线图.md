# 08_整体开发框架与开发路线图

## 1. 文档定位

本文档是 `xhs-automation` 项目的总开发框架与路线图，用于让 Codex、开发者和业务负责人在后续开发中始终保持一致。

本文档优先级低于 `AGENTS.md` 中的强制禁止事项，但高于临时聊天讨论。任何开发任务开始前，应先读取：

1. `AGENTS.md`
2. `docs/00_项目总览.md`
3. 本文档
4. 当前任务相关的专项文档
5. 当前任务相关的 `.agents/skills/*/SKILL.md`

## 2. 项目总目标

本项目目标是搭建一个以小红书为第一落地点、未来可扩展到 TikTok、Amazon、Shopee、独立站等渠道的自动化工作框架。

当前第一阶段重点是：

- 小红书热点追踪
- 小红书浏览器模拟发布
- 飞书多维表运营协同
- n8n 工作流编排
- browser-worker 浏览器执行层
- PostgreSQL 结构化底账
- MinIO 截图、素材、证据归档
- Codex / GitHub 驱动的可控开发流程

## 3. 强制开发边界

以下边界不可被任何任务、任何 Skill、任何临时提示覆盖：

1. 不允许调用小红书未授权接口。
2. 不允许逆向页面接口请求。
3. 不允许伪造 XHR、fetch、签名、Cookie 请求。
4. 不允许绕过登录、扫码、验证码、短信验证或二次验证。
5. 小红书发布和搜索必须通过真实浏览器页面模拟执行。
6. 遇到登录、验证码、风控、账号限制，必须暂停并返回人工处理状态。
7. 不允许把账号、密码、Cookie、Token、密钥写入代码仓库。
8. 所有截图、证据、异常现场必须可追踪。
9. 所有任务必须有 `job_id`。
10. 修改数据库结构必须生成 migration 或 schema 变更文件。
11. Codex 每次只允许完成一个明确 Task，不允许一次性开发整套系统。

## 4. 总体系统架构

核心链路：

```text
Feishu 多维表
→ n8n 定时或人工触发
→ browser-worker 接收 job
→ Browser Provider 启动或连接浏览器环境
→ Selenium / Playwright 控制真实页面
→ 截图与证据上传 MinIO
→ 结构化数据写入 PostgreSQL
→ 执行结果回写 Feishu
→ OpenClaw / 飞书机器人通知人工
```

## 5. 组件职责边界

### 5.1 Feishu 多维表

定位：运营工作台。

负责：

- 关键词池
- 热点池
- 内容发布池
- 人工审核状态
- 发布状态展示
- 异常处理备注
- 运营人工选择是否生成内容

不负责：

- 不作为唯一数据库
- 不保存完整执行日志
- 不保存大量截图原始文件
- 不直接控制浏览器

### 5.2 n8n

定位：流程编排层。

负责：

- 定时读取 Feishu 关键词池
- 读取待发布记录
- 生成 `search_job` / `publish_job`
- 调用 browser-worker API
- 接收执行结果
- 写入 PostgreSQL
- 回写 Feishu
- 处理异常分支

不负责：

- 不直接跑浏览器
- 不直接做复杂页面自动化
- 不保存大文件
- 不承担核心业务状态底账

### 5.3 browser-worker

定位：浏览器自动化执行层。

负责：

- 接收标准 job JSON
- 调用 Browser Provider
- 执行页面打开、输入、上传、截图等浏览器动作
- 返回结构化结果
- 记录执行日志
- 对异常现场截图

不负责：

- 不直接读取 Feishu
- 不调用 Coze / Dify
- 不做业务决策
- 不绕过验证码或风控
- 不伪造平台接口请求

### 5.4 Browser Provider

定位：浏览器环境适配层。

负责：

- 启动或连接具体浏览器环境
- 将 `account_id` 映射到浏览器 profile
- 返回 Selenium driver 或后续 Playwright context
- 屏蔽跨境卫士、本地指纹浏览器、普通 Chrome 的差异

第一阶段 provider：

- `selenium_chrome_provider`：本地开发调试用普通 Chrome

后续 provider：

- `kuajingvs_provider`：跨境卫士浏览器环境
- `local_fingerprint_provider`：本地指纹浏览器环境
- `manual_provider`：人工兜底发布包

### 5.5 PostgreSQL

定位：结构化底账。

负责：

- job 状态
- 搜索结果
- 发布结果
- 执行日志
- 错误码
- Feishu record 映射
- MinIO object key 映射

不负责：

- 不保存图片和截图二进制
- 不作为运营主界面

### 5.6 MinIO

定位：素材与证据仓库。

负责：

- 发布图片
- 搜索结果截图
- 详情页截图
- 发布前截图
- 发布结果截图
- 异常截图
- error.json

规则：

- 所有文件必须绑定 `job_id`
- 数据库只保存 MinIO object key 或 URL
- Feishu 可展示附件或外链，但不是主素材仓库

### 5.7 OpenClaw / 飞书机器人

定位：人机协同入口。

负责：

- 通知待审核
- 通知发布成功 / 失败
- 通知人工验证
- 查询任务状态
- 触发重试、继续、取消

不负责：

- 不直接发布
- 不直接跑浏览器
- 不保存系统底账

### 5.8 Coze / Dify

定位：智能分析与内容生成层。

MVP 阶段可使用 Coze，后续可替换为 Dify。

负责：

- 内容生成
- 热点评分
- 趋势分析
- 推荐角度
- 标题、正文、标签建议

不负责：

- 不直接发布
- 不直接推进状态
- 不直接控制浏览器
- 不直接写最终底账

## 7. 标准 Job 协议方向

### 7.1 publish_job

发布任务由 n8n 从 Feishu 发布池读取并生成。

核心字段：

- `job_id`
- `feishu_record_id`
- `account_id`
- `provider_type`
- `title`
- `body`
- `tags`
- `images`
- `publish_mode`
- `scheduled_at`

注意：

- `images` 第一阶段可使用 URL 列表
- 后续可升级为包含 `seq`、`role`、`url` 的结构化数组
- 图片顺序必须由结构化字段控制，不依赖飞书附件显示顺序

### 7.2 search_job

搜索任务由 n8n 从 Feishu 关键词池读取并生成。

核心字段：

- `job_id`
- `feishu_record_id`
- `account_id`
- `provider_type`
- `keyword`
- `keyword_type`
- `limit`
- `capture_screenshot`

### 7.3 worker_result

所有 browser-worker API 都应返回统一结果结构。

核心字段：

- `job_id`
- `status`
- `message`
- `error_code`
- `error_message`
- `screenshot_url`
- `note_url`
- `items`

常用状态：

- `accepted`
- `success`
- `failed`
- `waiting_human_verification`

## 8. 开发阶段路线图

### 阶段 0：项目规则层

目标：让 Codex 能稳定理解项目。

已完成：

- `AGENTS.md`
- `docs/00-07`
- `.agents/skills/xhs-hot-tracker`
- GitHub 仓库
- VS Code + Codex 连接

### 阶段 1：browser-worker 基础骨架

目标：先建立可运行服务，不接平台。

任务：

1. Task 01：FastAPI `/health`
2. Task 02：Pydantic Schemas
3. Task 03：Browser Provider 基类
4. Task 04：本地 `selenium_chrome_provider`
5. Task 05：统一错误码与结构化日志
6. Task 06：任务状态内存存储或轻量 job registry

验收：

- 本地服务可启动
- `/health` 可访问
- Schema 可 import 校验
- Provider 接口稳定
- 不涉及小红书页面

### 阶段 2：本地浏览器能力验证

目标：确认 browser-worker 能启动、截图、关闭普通 Chrome。

任务：

1. 启动本地 Chrome profile
2. 打开空白页或本地测试页
3. 截图保存到本地目录
4. 关闭浏览器
5. 输出结构化日志

验收：

- 不访问小红书
- 不访问外部网站
- 截图本地保存成功
- close_profile 能释放 driver

### 阶段 3：搜索任务最小闭环

目标：先完成小红书热点追踪 MVP。

任务：

1. `POST /api/xhs/search`
2. 接收 `SearchJob`
3. 打开小红书搜索页
4. 输入关键词
5. 截取搜索结果页
6. 提取前 N 条可见结果
7. 返回 `WorkerResult`
8. 本地保存截图，后续接 MinIO

验收：

- 浏览器实际页面顺序即排名依据
- 每条结果包含 rank
- 页面异常必须截图
- 不调用未授权接口

### 阶段 4：MinIO 与 PostgreSQL 接入

目标：把执行结果从本地调试升级为可追踪底账。

任务：

1. MinIO service
2. screenshot service
3. PostgreSQL schema
4. execution_logs
5. xhs_search_jobs
6. xhs_search_results
7. asset_files

验收：

- 截图写入 MinIO
- 数据库保存 object key
- 每个 job 有执行日志

### 阶段 5：n8n 与 Feishu 联调

目标：让运营从飞书触发和查看结果。

任务：

1. Feishu 关键词池读取
2. n8n 生成 `search_job`
3. n8n 调 browser-worker
4. 结果写 PostgreSQL
5. 回写 Feishu 热点池
6. 错误分支通知

验收：

- 飞书关键词池能驱动搜索
- Feishu 热点池能看到结果
- 失败能回写错误原因

### 阶段 6：发布任务最小闭环

目标：实现低频、可审计、可人工介入的小红书发布准备与浏览器模拟执行。

任务：

1. `POST /api/xhs/publish`
2. 接收 `PublishJob`
3. 下载图片到本地
4. 打开发布页
5. 上传图片
6. 填写标题、正文、标签
7. 发布前截图
8. 点击发布
9. 返回成功、失败或等待人工验证

验收：

- 遇到登录、验证码、扫码，必须暂停
- 失败必须截图
- 不调用未授权接口
- 不逆向请求
- 不高频并发发布

### 阶段 7：跨境卫士 Provider

目标：把本地 Chrome provider 替换为跨境卫士 provider。

任务：

1. 确认跨境卫士是否提供 profile 启动 API
2. 确认是否返回 WebDriver 或 CDP 地址
3. 实现 `kuajingvs_provider`
4. account_id 映射 profile_id
5. 支持失败回退到 manual provider

验收：

- xhs_publish_core 不依赖跨境卫士细节
- 切换 provider 不改业务流程
- profile 启动失败有错误码

### 阶段 8：OpenClaw / 飞书人机协同

目标：让人工处理异常、继续、重试、取消。

任务：

1. waiting_human_verification 通知
2. failed 通知
3. success 通知
4. resume webhook
5. retry webhook
6. cancel webhook

验收：

- 人工能收到明确任务信息
- 人工能知道该处理哪个 job
- 人工处理后可继续执行或重试

### 阶段 9：Coze / Dify 智能层

目标：把热点结果和内容生产联动。

任务：

1. 热点结果摘要
2. Coze 热点评分
3. 推荐角度生成
4. 内容草稿生成
5. 飞书待审核记录创建
6. 后续可迁移到 Dify

验收：

- Coze 只输出结构化 JSON
- n8n 负责写 Feishu / PostgreSQL
- Coze 不直接控制业务状态

## 9. Codex 使用规则

每次给 Codex 的任务必须满足：

1. 只完成一个 Task。
2. 必须说明允许修改哪些文件。
3. 必须说明禁止修改哪些文件。
4. 必须说明不允许实现的内容。
5. 完成后必须说明修改文件、测试方法、人工确认点。
6. 人工测试通过后再 commit。
7. commit 后再进入下一任务。

推荐流程：

```text
给 Codex 下 Task
→ Codex 修改文件
→ git status 检查范围
→ 本地测试
→ git add 精确文件
→ git commit
→ git push
→ 再进入下一 Task
```

## 12. 最终开发口径

本项目不是“接口逆向发布系统”，而是一个可审计、可暂停、可人工介入的浏览器自动化运营系统。

第一阶段目标不是追求全自动，而是先跑通：

```text
Feishu 输入
→ n8n 编排
→ browser-worker 执行
→ MinIO 留证
→ PostgreSQL 底账
→ Feishu 回写
→ 人工可控
```

所有后续开发都必须围绕这个闭环小步推进。
